# Database Data Definition Language (DDL)

This directory contains the database schematics and the ETL pipeline for our final project: `anime_analytics`. The SQL queries are housed in `ddl.py` as Python strings, dividing the pipeline into logical, manageable steps.

## Purpose

The main objective of `ddl.py` is to define the structure of the database, load raw scraped data, clean and normalize it, and distribute it into a relational database schema optimized for analytical queries.

## Pipeline Steps Explained (`ddl.py`)

### Step 1: Database Initialization
Creates a bare-bones database named `anime_analytics` (`str1`). 

### Step 2: Raw Import Tables
Defines the structure for "staging" tables: `raw_anime`, `raw_users`, and `raw_user_anime_list` (`str2`).
All columns are deliberately defined as `TEXT` to minimize ingestion errors when bulk-inserting from raw CSV files.

### Step 4: Final Normalized Schema Setup
Initializes our final, fully-normalized relational database schema (`str4`):
*   **Fact Tables / Main Entities:** `anime`, `users`, `user_anime_list` (a junction representing user interactions with anime).
*   **Dimensions:** Multi-valued anime metadata that were originally comma-separated strings are normalized into discrete dimensional tables: `genres`, `studios`, `producers`, `licensors`.
*   **Lookups:** `anime_list_status` handles fixed statuses (watching, completed, etc.).
*   **Junction Tables:** Maps the many-to-many relationship between anime and dimensions (e.g., `anime_genres`, `anime_studios`).
*   **Data Integrity:** Implements strong types (`INTEGER`, `NUMERIC`, `DATE`, `TIMESTAMP`, `BOOLEAN`), `PRIMARY KEY`s, `FOREIGN KEY`s with cascading updates, and `CHECK` constraints to ensure data quality.

### Step 5: The ETL Pipeline (Data Transformation)
This is where the magic happens (`str5`). It's wrapped in a single PostgreSQL transaction (`BEGIN ... COMMIT`) to ensure atomicity.
1.  **Helper Functions:** Defines PL/pgSQL functions (`safe_numeric`, `safe_int`, `safe_bool`, `safe_date`, `safe_timestamp`) to safely handle parsing and casting errors generated by "dirty" data (such as malformed dates like `0000-12-09`).
2.  **Basic Populations:** Populates lookup and primary entity tables (`anime`, `users`) with casted data from the raw tables.
3.  **Deduplication:** Uses the `DISTINCT ON (user_id, anime_id)` clause to fetch only the latest update row for `user_anime_list`, removing potential duplicates.
4.  **Array Unnesting:** Extracts comma-separated domains from the raw anime data (e.g., splitting a string of genres) using `regexp_split_to_table`, inserting the distinct tokens into dimension tables (`genres`), and mapping them through junction tables (`anime_genres`).
5.  **Indexing:** Creates multiple indexes automatically to boost the performance of future queries, such as indexing by status, genre, user interactions, or anime types.

### Step 6 & Reset
*   **Step 6 (`str6`)**: Executes sanity checks by running `COUNT(*)` queries on the final tables. This is helpful to rapidly verify data volumes.
*   **Reset (`str_reset`)**: Truncates all tables in the final schema, allowing the ETL process to be easily retried.

## Thoughts & Considerations

*   **Database Engine:** The DDL uses features specific to PostgreSQL, like `plpgsql` language, sequence generation (`GENERATED ALWAYS AS IDENTITY`), array parsing (`regexp_split_to_table`), and `DISTINCT ON` clauses. 
*   **Idempotency:** The process is designed so that the initial clearing (truncation) allows for you to run and re-run Step 5 continuously without suffering duplicate entries.
